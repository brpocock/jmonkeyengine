The 0.2 series of the exporter introduced detailed support for Skin & Bones. The 0.3 series greatly extended and improved animation support, and introduced many game integration features.

The 0.4 series has improved and rounded out material and texture support, including support for multiple Materials per Blender Object.

Change List

  • Changes since 0.3f
    1. Write morphInfluences for Armature Objects. This is a great way to centrally manage multiple morphs for an Armature assembly.
    2. Proper support for Blender texture image flag settings MipMap and Interpol. Until now we required both toggles to be set, but, sadly, still left both features unset in the exports.
    3. Implemented proper support and validation of Material Indexes. I.e. support multiple materials per Blender Object.
    4. Support for multiple texture channels. Each such channel may do per-face or sticky vert UV mapping.
    5. Support for multiple uv layers.
    6. OB and ME type Material links are now translated according to robust rules which can be generalized to: Mats are translated to RenderStates on a TriMesh or QuadMesh, except for normal Blender Mesh Objects with a single (Material Index) OB Material Link. Node Render States can not be applied properly to other situations.
    7. Partial implementation of FAVORJOIN mode, where jME Nodes are created only where necessary, instead of the default behavior to mirror Blender as close as possible.
    8. [Significantly missing from release 0.4a are documentation updates.]
  • Changes since 0.3e
    1. Eliminated requirement for Bone Heat. Apply bone weights using your favorite method for now on.
    2. Bugs fixed which caused corruption on the jME side when some skin vertexes were not weighted.
    3. Support skin & bones weightings where some verts have no weights.
    4. Class files excluded from distros (they added an unnecessary tight binding/dependency between HottBJ version and jME version).
  • Changes since 0.3d
    1. Output model files now renderer-independent.
    2. Fixed armature export bug.
  • Changes since 0.3c
    1. Improvements to pre-validation of animation setup.
    2. State restoration is now reliable.
    3. Write SkinNode components with relationships exactly matching that used by jME natively.
    4. Game properties of format "jme.*" stripped from exported UserData in a general and efficient way. These properties are for internal use of the exporter. Users should use their own name prefix, or none.
    5. Added support for jME's new SkinTransferNodes, for writing skin meshes which can be swapped in and out at jME runtime.
    6. Added support for jME skin regions, by use of game properties named "jme.skinRegion" or object names with suffix @regionName.
    7. Fixed poor formatting when hash value for Texture exceeds Python data type's value range.
    8. Since a Rest Pose Frame is required for skin assimilation in jME, the exporter now looks for one and identifies the best one to the AnimationController.
    9. Since Actions are always written with Armatures, I've switched the corresponding AnimationController to be inactive. Easy enough for the developer to set to active when needed.
    10. Per-Action frame names are now transferred and used by jME's BoneAnimation. For now, user must set a game property of name jme.frameNames.YOUR_ACTION_NAME, but want to use Blender's pose markers automatically in the future.
    11. Support for new jME morphing feature.
    12. Wrote new Online Help page about HottBJ's game integration features.
  • Changes since 0.3b
    1. One armature may now skin multiple deforming meshes (via parenting and/or modifier associations).
    2. Fixed bug with texture sharing which sometimes caused some textures to be replaced with references to others in the same export.
    3. Added rigorous checking for bone scales which could produce inaccurate exports.
    4. Fixed logic bug which somes resulted in valid animation channels being dropped.
    5. Fixed bug which broke export of non-skinned bone animations.
  • Changes since 0.3a
    1. Objects in non-selected Blender layers are automatically excluded from exports, regardless of all-in-scene or selected-item mode.
    See Subversion log for history before the 0.3 series.
